# Security Policy

## Threat Model

Fluxo is a headless wallet core. It protects encrypted vault material against offline brute-force and tampering under the assumptions below:

- the vault password has meaningful entropy;
- the vault JSON is stored by the integrating application without modification;
- signing happens through short-lived runtime sessions;
- private keys and mnemonics are not logged or persisted by the host application.

Fluxo does not protect against a compromised operating system, malicious host application, keyloggers, screen capture, memory inspection, supply-chain compromise, or weak user passwords. It is not a hardware wallet and it is not "unhackable."

## Security Boundaries

The repository intentionally avoids high-risk application surfaces:

- no bundled interface;
- no application shell;
- no dapp provider injection;
- no content scripts;
- no cloud sync;
- no transaction broadcasting;
- no hosted custody.

Integrators own the UX, storage, RPC, telemetry, backup, lock, reset, and transaction-review layers.

## Secret Handling

- New wallets use a BIP39 seed phrase generated by the Go wallet core.
- Vault v3 encrypts HD mnemonics with XChaCha20-Poly1305 and a 24-byte nonce.
- The encryption key is derived with Argon2id using 256 MiB memory, 4 passes, `p=1`, a 32-byte salt, and a 32-byte key.
- Vault metadata is authenticated as AEAD additional data.
- KDF downgrade below the v3 defaults is rejected.
- New-wallet creation returns the generated mnemonic only once for user backup.
- Import and unlock flows do not return mnemonic or private key material.
- Message signing is performed by `sessionId`.
- Sessions have a default 5-minute TTL and can be explicitly locked.
- Legacy v1 vaults are supported only for unlock-and-migrate to legacy private-key v2.

Go and WASM memory cannot provide hardened zeroization guarantees. Treat any process that hosts an unlocked session as sensitive.

## Vault v3 Format

The persisted object has this high-level shape:

```json
{
  "header": {
    "version": 3,
    "cipher": "XCHACHA20-POLY1305",
    "kdf": "ARGON2ID",
    "kind": "hd-mnemonic",
    "kdfParams": {
      "memoryKiB": 262144,
      "passes": 4,
      "parallelism": 1,
      "saltBytes": 32,
      "keyBytes": 32
    },
    "address": "0x...",
    "createdAt": "..."
  },
  "salt": "...",
  "nonce": "...",
  "ciphertext": "..."
}
```

The `header` is authenticated as AEAD additional data. Changing the address, cipher, KDF parameters, nonce, or ciphertext causes unlock failure.

## Integration Requirements

Applications using Fluxo should:

- store only encrypted vault JSON;
- never persist session IDs as durable credentials;
- avoid logging passwords, mnemonics, private keys, vault plaintext, signatures, or raw session state;
- clear one-time mnemonic backup data from UI state after the user completes backup;
- lock sessions on app background, tab close, logout, or inactivity;
- make RPC endpoint choices explicit to users because balance reads reveal addresses to providers;
- add independent review before enabling transaction signing or broadcast flows;
- use hardware wallets for high-value custody.

## Limitations

- Password strength still matters. Argon2id slows offline guessing; it does not make weak passwords safe.
- A malicious host app can misuse any wallet core it embeds.
- Public RPC reads can leak address and chain metadata to providers.
- No transaction broadcast, dapp provider, content script, or permission system exists in this scope.

## Reporting Vulnerabilities

Please report vulnerabilities privately before public disclosure.
